Ember Control Pipe

__word__ pattern: control messages handled in-process, never forwarded to LLM

App sends prompt
Raw UTF-8 over QUIC stream
↓
Is __word__ ? (exact match)
__get_style__ | __fetch_push__ | __check_in__ | __whatever__
↓ Yes
Control pipe (in-process)
__get_style__ → read CSS, return
__fetch_push__ → long-poll queue, return payload
__check_in__ → queue has msg? stream it : synthetic → LLM
__whatever__ → return empty stream
↓ No (user prompt)
Forward to LLM
format_prompt → gRPC complete_stream → stream tokens back

App → Server Control Messages

Message Server Action Reaches LLM?
__get_style__ Read CSS file, return immediately No
__fetch_push__ Pop from proactive_queue (long-poll 60s) No
__check_in__ If queue has msg → stream it; else synthetic prompt → LLM Only synthetic
__whatever__ Any other __word__ → return empty stream No

AI → App (Control Pipe)

The AI can output control commands. Server strips them from the stream and queues for app.

AI Output Server Action App Receives
__app_clear__ Strip, queue "__app_clear__" Next __fetch_push__ → applyPushPayload → reinitializeDisplay
<ember_push>app clear</ember_push> Strip, queue "app clear" Same
<ember_push>{"chat":[...]}</ember_push> Strip, queue JSON applyPushPayload → replace chat

Pattern Rules

__ + word (alphanumeric + underscore) + __
Examples: __get_style__, __fetch_push__, __app_clear__, __refresh__

The AI is instructed to never say, speak, or echo __word__ tokens. They are protocol, not content.
